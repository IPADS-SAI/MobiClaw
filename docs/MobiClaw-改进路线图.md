# MobiClaw 改进路线图（对标 OpenClaw / NanoClaw）

> 目标：把 MobiClaw 从“可运行的多 Agent 编排原型”升级为“可持续运行、可扩展、可评估、可安全落地”的个人助理框架，重点强化自主规划、跨工具协作、手机执行闭环与长期记忆能力。

---

## 0. 当前状态与优先改进切入点

结合当前实现，系统已具备基础闭环：

- 编排入口与多模式运行（demo / interactive / daily / agent_task）；
- Steward + Worker 双 Agent 协作；
- MobiAgent 手机操作网关；
- 基础 Daily 任务执行与 run_id 事件日志。

但离“个人助理框架”还有关键缺口：

1. **任务规划能力偏弱（无显式 Planner/Executor/Reviewer 分层）**：当前以单 ReAct 为主，复杂任务可解释性与稳定性不足。
2. **异步任务可靠性不足**：任务状态保存在内存，进程重启会丢失；缺少统一重试、幂等键和任务恢复机制。
3. **手机任务验证仍偏启发式**：主要基于 OCR/reasoning 文本匹配，缺少结构化状态机与多证据一致性校验。
4. **知识库利用深度不足**：已能 add/search/rag，但尚未形成“长期记忆 + 用户画像 + 偏好/日程/联系人”稳定 schema。
5. **缺少系统评测基线**：暂无“任务成功率、工具成功率、时延、成本、错误分类”指标体系和回归集。
6. **安全治理缺口**：对敏感动作、权限边界、审计追踪、数据脱敏的机制仍可加强。

---

## 1. 总体目标架构（建议）

建议将现有架构升级为“六层能力栈”：

1. **Intent & Planning 层**：用户意图解析、任务拆解、约束提取、计划生成（可修订）。
2. **Execution Orchestration 层**：步骤调度、并行/串行、失败重试、超时管理、恢复点。
3. **Tool & Device 层**：MobiAgent、Web、Shell、File、第三方 API 统一抽象。
4. **Memory & Knowledge 层**：本地长期记忆、会话摘要、事实库、用户画像与任务历史。
5. **Evaluation & Safety 层**：在线监控、离线评测、策略护栏、敏感操作确认。
6. **Product Gateway 层**：同步/异步任务 API、事件流、可观测仪表板。

## 2. 分阶段改进路线（详细）

> 节奏建议：12~16 周，按 P0/P1/P2 递进，优先做“可用性 + 稳定性 + 可评估”。

### Phase P0（第 1~4 周）：稳定性与可观测性打底

#### P0-1 任务基础设施升级（最高优先）

- 将 `gateway_server` 的 `_JOB_STORE` 从内存迁移为持久层（SQLite/PostgreSQL/Redis 均可）。
- 为任务引入：
  - `idempotency_key`（避免重复提交）；
  - `status` 状态机（queued/running/waiting_user/completed/failed/cancelled）；
  - `retry_policy`（指数退避 + 最大次数 + 可重试错误类型）。
- 新增任务恢复能力：服务重启后扫描 `running` 任务并恢复或标记。

**验收标准**：重启服务后，任务状态不丢失；重复请求可安全去重。

#### P0-2 统一观测与审计

- 将当前 run 日志扩展为结构化 trace（trace_id/span_id/tool_call_id）。
- 统一记录：prompt、工具参数摘要、耗时、结果、错误码、token 用量。
- 建立最小指标面板（Prometheus + Grafana 或轻量日志聚合）：
  - 任务成功率、工具成功率、P95 时延、重试率、人工介入率。

**验收标准**：可按 run_id 回放完整链路，并定位失败步骤。

#### P0-3 手机执行可靠性增强

- 将“成功判定”从文本启发式升级为“证据策略”：
  - OCR 命中 + UI 元素命中 + action 序列一致性 >= 阈值；
  - 输出统一 `verification_report`（通过/失败/不确定）。
- 在 Mobi 任务模板中定义“可验证目标”（例如：页面标题、关键字段值、截图区域）。

**验收标准**：关键手机任务（消息读取、提醒创建）成功率显著提升，且失败可解释。

---

### Phase P1（第 5~10 周）：能力增强与多 Agent 协同

#### P1-1 引入三角色协作流（Planner / Executor / Reviewer）

在现有 Steward/Worker 基础上演进：

- **Planner**：拆解目标、生成步骤计划和验收条件；
- **Executor**：调用工具执行单步并产出证据；
- **Reviewer**：验证步骤完成度并决定继续/回滚/请求用户确认。

保留 Steward 作为“总控协调器”，负责状态推进与用户沟通。

**验收标准**：复杂任务（>5 步）可展示“计划-执行-校验”全过程。

#### P1-2 Tool Contract 标准化

- 为所有工具定义统一输入/输出 schema（Pydantic），明确：
  - 必填参数、默认值、错误码、可重试性；
  - 结果中的 `evidence`, `confidence`, `next_actions`。
- 引入工具注册中心（capability registry），按场景动态启用工具。

**验收标准**：Agent 能稳定读取结构化返回，不再依赖自由文本解析。
#### P1-3 Agentscope 协同能力体系化

- 使用 Agentscope 的 agent 定制能力建立“角色模板库”：
  - 日程助理、信息检索助理、购物执行助理、风险提醒助理。
- 增加“群体协作策略”：
  - 竞争式（多个 worker 并行检索，reviewer 选优）；
  - 流水线式（检索->抽取->决策->执行）。

**验收标准**：在同类任务上，多 agent 协作优于单 agent 成功率/时延。

---

### Phase P2（第 11~16 周）：产品化与生态扩展

#### P2-1 用户确认与策略引擎

- 引入策略 DSL（或 YAML 规则）：
  - 何时必须二次确认（转账、发消息、下单）；
  - 何时允许自动执行（仅查询、只读分析）。
- 支持“组织/家庭模式”多租户策略隔离。

#### P2-2 自动化工作流市场化

- 提供 Workflow 模板：
  - 每日摘要、会议纪要整理、票据收集报销、社交通知优先级排序。
- 支持“可视化编排 + 代码编排”双模式。

#### P2-3 评测与持续优化闭环

- 建立 benchmark 套件：
  - 手机任务集（UI 变化、弱网、权限受限）；
  - 知识检索集（多跳问题、冲突事实）；
  - 端到端任务集（计划、执行、复盘）。
- 每次版本发布执行回归并输出 scorecard。

#### P2-4 多端扩展

- 除手机外，扩展到桌面自动化、浏览器插件、IM 机器人接口。
- 统一事件总线（WebSocket / queue）承载异步状态推送。

## 3. 关键技术方案建议（落地视角）

### 3.1 任务模型（建议字段）

- `task_id`, `user_id`, `intent`, `constraints`, `priority`, `deadline`
- `plan`（steps + success_criteria）
- `state`（当前阶段）
- `artifacts`（截图、OCR、日志、文件）
- `decision_log`（关键决策点）
- `result`, `postmortem`

### 3.2 失败分类标准

统一错误分类，便于自动恢复：

- `TOOL_TIMEOUT`, `TOOL_AUTH`, `TOOL_SCHEMA`, `DEVICE_NOT_READY`, `KB_UNAVAILABLE`, `PLAN_INVALID`, `POLICY_BLOCKED`。

并映射“自动动作”：重试/切换工具/请求用户输入/终止并上报。

### 3.3 计划与执行分离

- 规划阶段只产出计划，不做工具调用；
- 执行阶段严格按计划步骤推进；
- 复核阶段独立评估，避免“自说自话”偏差。

### 3.4 知识治理

- 记忆写入前做去重和冲突检测；
- 高敏感信息加密或脱敏存储；
- 定期进行“记忆体检”（过期、冲突、低置信清理）。

## 4. 优先级 Backlog（可直接用于排期）

### P0 必做（立即开始）

1. 持久化 Job Store + 状态机 + 幂等键。
2. 全链路结构化日志与基础指标。
3. Mobi 执行验证报告标准化（verification_report）。
4. 工具错误码与重试策略统一。

### P1 强化（第二阶段）

1. Planner/Executor/Reviewer 协作流。
2. Agentscope 多角色模板与协作策略。
3. 任务复盘与失败学习自动写回知识库。

### P2 增长（第三阶段）

1. 策略引擎与敏感操作治理。
2. 工作流模板市场 + 可视化编排。
3. 基准评测平台与版本 scorecard。
4. 多端能力扩展（桌面/浏览器/IM）。

---

## 5. 里程碑与量化目标（建议）

### M1（4 周）

- 任务可持久化与恢复；
- 基础监控上线；
- 手机任务成功率提升到可观测且可解释。

### M2（10 周）

- 多 Agent 协作流上线；
- 复杂任务成功率相比基线提升 25%+。

### M3（16 周）

- 策略/安全/评测闭环成型；
- 对外提供模板化自动助理能力；
- 达到“可长期运行的个人助理框架”标准。

---

## 6. 与现有代码的衔接建议（最小改造路径）

1. 在不破坏现有入口（`app.py` / `workflows.py`）前提下，先替换 `gateway_server` 的任务存储与状态机。
2. 保留 Steward 作为总控，新增 Planner/Reviewer 作为可选模式开关（灰度发布）。
3. DailyTasks 先升级为"可计划 + 可回放 + 可复盘"的批处理框架，再逐步接入实时触发。

> 以上路线可确保“先稳住可用性，再做能力跃迁”，避免一次性重构导致系统不可用。
